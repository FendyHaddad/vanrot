# Phase 15A Route Contract Production

## Purpose

Phase 15A makes the router route contract production-ready while preserving the Vanrot rule that user-facing templates and pages do not repeat route path strings, route label strings, or route-name strings.

This phase focuses on route definitions, typed params, query strings, route-link generation, breadcrumbs, active links, and diagnostics. It does not implement nested outlets, layout shells, guards, redirects, or preloading. Those remain later Phase 15 slices.

## Source Of Truth

`src/routes.ts` remains the owning source of truth for route paths, route labels, query metadata, and breadcrumb metadata.

Route path and label strings are acceptable only inside `src/routes.ts`, because that file owns the route contract. User pages, templates, CSS, and app code should reference route objects and framework primitives instead of repeating those strings.

Vanrot-authored templates continue to use the approved Phase 8 route primitive:

```html
<vr route.productDetail param.productId={product.id} />
<vr-breadcrumbs />
```

They should not use route-name strings, path strings, label strings, active-class strings, or custom CSS hook strings.

## Route Definition Model

Phase 15A adds `createRoutes()` builder refs that are normalized by `defineRoutes(...)`. The existing object form remains valid for Phase 8 compatibility.

```ts
const routes = createRoutes();

const shop = routes.page({
  path: '/shop',
  label: 'Shop',
  breadcrumb: routes.breadcrumb.root(),
  page: ShopPage,
});

const product = shop.page({
  path: 'product',
  label: 'Products',
  breadcrumb: routes.breadcrumb.parent(shop),
  page: ProductListPage,
});

const productDetail = product.page({
  path: ':productId',
  label: 'Product detail',
  breadcrumb: routes.breadcrumb.parent(product),
  page: ProductDetailPage,
});

export const route = defineRoutes({ shop, product, productDetail });
```

The builder form allows parent routes and breadcrumb parents to be expressed as route object references, not route-name strings. Child paths are relative when a parent is present, and the router normalizes the full path. In the example above, `productDetail` resolves to `/shop/product/:productId`.

The route object returned by `defineRoutes()` remains the value exposed to app components:

```ts
import { route as appRoute } from '../routes';

export class AppComponent {
  route = appRoute;
}
```

## Params And Query Strings

Route params are inferred from path segments such as `:productId`. The compiler validates `param.*` bindings on `<vr route.name />` against the selected route.

```html
<vr route.productDetail param.productId={product.id} />
```

Query metadata is declared in `src/routes.ts` when a route has known query values:

```ts
const routes = createRoutes();

const search = routes.page({
  path: '/search',
  label: 'Search',
  page: SearchPage,
  query: {
    term: { default: '' },
    tags: { array: true },
  },
});
```

Templates bind query values through the same route primitive:

```html
<vr route.search query.term={searchTerm} query.tags={selectedTags} />
```

Phase 15A owns encoding, array query values, defaults, and history behavior for generated URLs. Query reads should not break path param matching.

## URL Building And Navigation

Router links, navigation, breadcrumbs, tests, and future helpers all use one internal URL builder. The builder accepts a route object plus params and query values, then returns the encoded href.

This keeps behavior consistent across:

- `<vr route.name />`
- breadcrumb links
- runtime `navigate(...)`
- tests and future testing helpers

The URL builder handles:

- parent path composition
- missing required params
- unknown params
- query encoding
- array query values
- query defaults
- browser history updates

## Route Links

The compiler continues to lower `<vr route.name />` into router helper calls. Phase 15A adds support for `param.*` and `query.*` bindings.

The generated anchor uses:

- href from the internal URL builder
- text from route metadata
- accessible native anchor behavior
- framework-owned active state

Templates do not provide inline route labels, route-name strings, path strings, active-class strings, or CSS hook strings.

## Active State

Active state is framework-owned.

When a generated route link matches the current route exactly, the anchor receives `aria-current="page"`. Parent route links can also be marked active when the current route descends from them, using framework-owned generated behavior.

Phase 15A does not require users to author active classes or CSS selectors in pages. If internal styling hooks are needed, they are generated by the framework and documented as framework output, not app-authored source.

## Breadcrumbs

Phase 15A adds breadcrumb metadata to route definitions and a framework-owned breadcrumb primitive.

```html
<vr-breadcrumbs />
```

Breadcrumb chains are built from route object references:

- root routes use `routes.breadcrumb.root()`
- child routes use `routes.breadcrumb.parent(parentRoute)`
- labels default to route labels
- hrefs use the same URL builder as route links

For `/shop/product/42`, the example route table produces:

```txt
Shop / Products / Product detail
```

Each crumb points to its route object. Current route params are reused when the crumb route needs params. Static breadcrumb labels are in scope for Phase 15A. Dynamic entity labels are deferred unless they can be expressed through stable route metadata without adding page-level string literals.

## Diagnostics

Phase 15A route diagnostics should be reusable by router, compiler, and CLI surfaces. Every diagnostic should include a stable code, message, source span when available, suggestion, and docs hook.

Diagnostics cover:

- duplicate full paths
- invalid parent path composition
- route missing `page` or `loadPage`
- invalid param names in route paths
- missing required `param.*` on `<vr route.name />`
- unknown `param.*`
- unknown `query.*`
- query binding without query metadata when strict route query checking is enabled
- breadcrumb parent not present in the returned route table
- impossible breadcrumb param resolution
- invalid route refs in templates

## Testing

Phase 15A should add focused failing tests before implementation for:

- builder-form `defineRoutes()` path normalization
- object-form `defineRoutes()` compatibility
- parent route references without route-name strings
- breadcrumb parent references without route-name strings
- typed param inference from path segments
- URL building with encoded params
- query defaults and arrays
- query strings not breaking route matching
- active link exact behavior
- active link ancestor behavior
- `<vr route.name param.* query.* />` compiler lowering
- `<vr-breadcrumbs />` compiler lowering and runtime behavior
- generated starter route strings and labels staying in `src/routes.ts`
- route diagnostics with code, message, source span, suggestion, and docs hook

The final verification command remains:

```sh
pnpm verify
```

## Deferred To Later Phase 15 Slices

Phase 15A does not implement:

- nested outlet rendering
- layout shell rendering
- route guards
- async guard cancellation
- redirects
- lazy-page preloading
- dynamic entity breadcrumb labels that depend on page data loading
- SSR-specific router behavior

Those features should be split into later Phase 15 sub-phases after the route contract is stable.

## Production Readiness Outcome

After Phase 15A, Vanrot should support production route contract authoring where:

- `src/routes.ts` is the single route source of truth
- route parent relationships use route object refs
- breadcrumbs use route object refs
- templates link to routes without route-name strings
- param and query links are generated without path strings
- active state is framework-owned
- route contract mistakes produce diagnostics
- existing Phase 8 router apps remain compatible
